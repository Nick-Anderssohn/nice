## 3. Why GPUI (if rewriting)

### Hard constraint first: native efficiency ‚Äî clean for the *framework*, **provisional for the recommended hybrid**
**GPUI the framework** clears "match or beat the current native Swift app" more cleanly than any other candidate: it is the exact stack Zed ships (compiled Rust, no GC, no JS runtime, GPU‚Äëcomposited via its own Metal/blade‚Äëgraphics pipeline), and the spike confirmed it builds and runs standalone on this machine (macOS 26.5.1, Apple silicon) with no native code written by us. **Path B inherits that Zed‚Äëclass efficiency directly.**

**But the *recommended* Path A is not "GPUI" ‚Äî it is a dual‚ÄëMetal‚Äëstack hybrid:** GPUI's blade‚Äëgraphics pipeline compositing chrome *over* SwiftTerm's bespoke Metal renderer, in one process, across an objc2 z‚Äëorder seam. That is precisely the "runs a second GPU stack" objection this report uses to mark **down** `flutter-reuse-swiftterm` and to caution `flutter-ffi` ‚Äî and honesty requires applying it to our own hybrid. **No FPS, latency, or memory has been measured for the dual‚Äëstack hybrid;** the spikes only confirmed that GPUI launches and that an NSView can be hosted under objc2 ‚Äî *not* that two live Metal stacks coexist at burst FPS inside the Swift baseline's memory envelope, and *not* through a live responder chain (the embed spike used a Swift **stub** view and invoked overridden selectors directly). **Path A's efficiency PASS is therefore provisional** (scored `5‚ÄÝ` in ¬ß5) and must be confirmed by Phase‚Äë0 against the current Nice baseline before commitment. If the hybrid regresses FPS or memory, the efficiency‚Äëclean options are **Path B** (single GPUI stack) or **objc2‚Äëhybrid** (single AppKit‚Äëowned tree, ¬ß4).

Below the hybrid, nothing changes: **Tauri (`tauri-xtermjs`) and Dioxus (`dioxus-alacritty`) remain eliminated** on this hard bar (WKWebView capped at 60 Hz on ProMotion + jitter under burst; Dioxus's only native path, Blitz, is alpha and explicitly "not performance‚Äëtuned"). **Flutter** passes but sits *slightly above* the Swift idle baseline (~30‚Äì40 MB engine/Skia base), and its reuse variant runs the same second‚ÄëGPU‚Äëstack risk.

### Primary driver: AI velocity ‚Äî D1 vs D2 honestly separated
- **D1 (language corpus):** Rust is top‚Äëtier for Claude (deep corpus + a borrow‚Äëchecker/`rustc` self‚Äëcorrection loop that beats Swift's slower, sometimes‚Äësilent SwiftUI failure mode). **GPUI scores D1 = 5.**
- **D2 (UI‚Äëframework corpus):** **GPUI is genuinely weak ‚Äî comparable to SwiftUI** (pre‚Äë1.0, Zed‚Äëinternal, churning API, thin public corpus). **D2 = 2.** Rust's D1 strength does **not** mask this.

So GPUI's AI‚Äëvelocity improvement over SwiftUI is **language‚Äëdriven, not framework‚Äëdriven.** Two mechanisms are often conflated here, and **only one drives Claude's default reliability**:

- **(a) Training‚Äëcorpus depth ‚Äî what actually shapes Claude's in‚Äëweights priors ‚Äî is genuinely thin and churning for GPUI (D2 = 2, unchanged from SwiftUI).** This is the number that governs how reliably Claude writes correct GPUI 0.7 chrome *by default*, and it does not beat SwiftUI.
- **(b) Dev‚Äëtime augmentation ‚Äî feeding current Zed / `gpui-component` source into context ‚Äî is real but weaker than the framing implies.** The entire Zed source is open, high‚Äëquality GPUI usage, and `longbridge/gpui-component` adds 60+ documented components; a developer or a retrieval‚Äëaugmented session can lean on them. **But** this requires the human to supply *current* sources, and because GPUI's API churns (the tree already moved `crates/gpui ‚Üí crates/gpui_macos`; 0.2.2 is ~8 months stale), fed‚Äëin current source can actively *conflict* with the model's stale in‚Äëweights snapshot. So (b) mitigates but does **not** lift the near‚Äëhard primary‚Äëdriver bar the way a deep native corpus (Flutter/React) would. **D2 stays 2.**

Net: the chrome becomes more AI‚Äëassistable **because of the language, not the framework** ‚Äî a defensible win that clears the near‚Äëhard bar, not the Electron/React‚Äëclass jump it is sometimes painted as.

### Why GPUI over Flutter (the genuinely stronger primary‚Äëdriver candidate)
**Flutter is the stronger two‚Äëaxis AI bet** (D1 = 4 Dart, **D2 = 5** ‚Äî one of the best‚Äëdocumented UI toolkits anywhere). It was weighed seriously. It loses to GPUI on three grounds:
1. It is weaker on the **hard** efficiency constraint (above) ‚Äî though note GPUI's own Path‚ÄëA efficiency is itself provisional.
2. Its huge D2 corpus is concentrated in **mobile/general widgets**; Nice's hardest chrome (vibrancy wiring, native traffic‚Äëlight geometry, NSDraggingSource tear‚Äëoff, multi‚Äëwindow) lives in Flutter desktop's *thin* corpus **and crosses into hand‚Äëwritten Swift/ObjC**, where Claude loses the Dart‚Äëanalyzer loop. The AI win is weakest exactly where Nice is hardest.
3. **Zero Swift is *not* a Path‚ÄëA differentiator.** Flutter keeps vibrancy, traffic‚Äëlight placement, tear‚Äëoff detection, and the close‚Äëconfirm delegate in a residual Swift/ObjC shim. **GPUI reaches zero Swift only on Path B** (rebuild the terminal on `alacritty_terminal`); the **recommended Path A keeps swiftc in the build permanently** (the `@_cdecl` shim + reverse‚ÄëFFI + cargo+swiftc two‚Äëtoolchain ‚Äî spike‚Äëconfirmed). So on Path A, *both* stacks keep Swift; zero‚ÄëSwift is a real GPUI edge **only in the Path‚ÄëB world**, which ‚Äî like Flutter‚ÄëFFI ‚Äî sacrifices the renderer reuse. Treat zero‚ÄëSwift as a tie‚Äëbreaker for Path B, not a virtue of the recommended hybrid.

### Where the spikes changed the picture
1. **GPUI is no longer "Zed‚Äëinternal, can't depend on it."** `gpui = "0.2.2"` is on crates.io, compiled standalone in ~36 s, ran first try. (`spike-gpui-glass`: `‚Ä¶/scratchpad/spike-gpui-glass/glassdemo`)
2. **Vibrancy + native traffic lights are built into GPUI with zero native code.** `WindowBackgroundAppearance::Blurred` inserts a real `NSVisualEffectView`; `TitlebarOptions.traffic_light_position` repositions the *real* native buttons ‚Äî refuting the desk claim that GPUI relies on a private CGS API. (`gpui-0.2.2/src/platform/mac/window.rs:1257‚Äë1311`, `:2500‚Äë2510`)
3. **`libghostty` is dead on arrival; `alacritty_terminal` is the proven Rust core.** `alacritty_terminal` + `portable-pty` ran a real `/bin/sh`, parsed the PTY stream, decoded per‚Äëcell SGR with zero build friction; `libghostty-vt` failed at `build.rs:365` (zig absent) and is parser‚Äëonly. (`spike-rust-term`; `ghostty-probe`)
4. **Hosting the SwiftTerm NSView under a Rust/objc2 host is mechanically solved ‚Äî but only at the stub/selector level.** A pure‚Äëobjc2 NSView and a Swift‚Äëcompiled NSView were hosted, made first responder, resized, and driven across a C ABI (~150 lines, PASS). **Crucially, the spike dispatched events by invoking overridden selectors directly, *not* through a live windowserver responder chain, and used a stub ‚Äî not the real SwiftTerm view** (`spike-reuse-swiftterm` "whatFailed"). The remaining cost is the FFI shim + two‚Äëtoolchain build **and** the unproven live‚Äëresponder/IME and z‚Äëorder seams (¬ß4, ¬ß10).

## 4. GPUI‚Äëhybrid vs objc2‚Äëhybrid: the near‚Äëtwin the report must not dodge

The recommended pick and the *rejected* `objc2-reuse-swiftterm` candidate are **nearly the same architecture.** Both embed the identical SwiftTerm Metal NSView; both keep the Swift toolchain under Path A; both reach Rust for the chrome; **both score D2 = 2.** The recommended GPUI hybrid is, structurally, *the objc2 candidate with GPUI swapped in for hand‚Äëwritten AppKit chrome.*

So the report's stated reason for rejecting objc2 ‚Äî "it fails the primary driver (D2 = 2, hand‚Äëwriting AppKit)" ‚Äî **cannot be the real differentiator, because GPUI fails that same framework‚Äëcorpus bar.** The honest head‚Äëto‚Äëhead:

| | GPUI‚Äëhybrid (recommended) | objc2‚Äëhybrid (`objc2-reuse-swiftterm`) |
|---|---|---|
| Framework corpus (D2) | **2** | **2** |
| Language safety (D1) | **5** (pure safe Rust) | 4 (unsafe `msg_send!` holes in the AppKit‚Äëtouching code) |
| Chrome authoring model | **Declarative** element/component tree ‚Äî better DX for the high‚Äëvolume chrome iteration that is the whole point | **Imperative**, verbose, hand‚Äëwritten AppKit through objc2; example‚Äëthin, no rich‚Äëchrome reference app |
| SwiftUI‚Äëfighting machinery | Gone (not SwiftUI) | Gone (not SwiftUI) |
| SwiftTerm NSView embed | **Unproven z‚Äëorder/transparency/responder seam** ‚Äî GPUI draws its *whole* content view itself; a foreign terminal view must sit at the right z‚Äëorder with GPUI compositing transparently over it | **Trivially proven (D5 = 5)** ‚Äî objc2 owns the entire NSView tree, so the terminal is just `addSubview`; **no seam** |
| Maturity (D7) | 2 (pre‚Äë1.0, single‚Äëvendor, churning) | 3 (objc2 is the de‚Äëfacto Rust‚ÜîApple standard, broadly adopted) |

**The real, defensible case for GPUI over objc2 is NOT "objc2 fails the primary driver" ‚Äî it is the declarative‚ÄëDX/maintainability axis plus full safe‚ÄëRust (D1 5 vs 4).** GPUI gives a declarative component framework for the high‚Äëvolume chrome (tabs, pills, settings, theming, file tree); objc2 makes you hand‚Äëwrite every bit of that imperatively in a niche, unsafe, example‚Äëthin dialect.

**But that edge is bought at a price the report must own:** GPUI *uniquely introduces* the z‚Äëorder/transparency/responder seam that is this report's single load‚Äëbearing risk (¬ß9, ¬ß10). objc2 *avoids that seam entirely* by owning the view tree. We are therefore recommending the option that carries its own biggest open risk over a sibling that doesn't ‚Äî justified **only** if (a) the declarative‚ÄëDX win outweighs the seam risk, and (b) Phase‚Äë0 retires the seam.

**Consequence for the fallback tree:** if Phase‚Äë0 fails *specifically on the GPUI z‚Äëorder/responder seam* (rather than on AppKit event embedding in general), the natural fallback is **objc2‚Äëhybrid** ‚Äî it keeps the proven renderer *and* avoids the seam, at the cost of hand‚Äëwritten imperative chrome ‚Äî **not only Path B.** objc2 is a first‚Äëclass fallback, not a dismissed also‚Äëran.

## 6. The liquid‚Äëglass verdict

**The signature look can be preserved ‚Äî and the spikes prove it is *not* a GPUI‚Äëspecific or AppKit‚Äëexclusive problem.**

## 8. Migration strategy

**Shape: hybrid (recommended) with a clean‚Äënative or objc2 fallback for the terminal layer.** The chrome is a full rewrite either way ‚Äî that is the goal. The decision fork is the *terminal layer* (¬ß4, ¬ß10).

### What of the Metal‚Äërenderer work is reusable, and the interop cost

**Path A ‚Äî REUSE (hybrid, recommended primary):** embed the existing SwiftTerm fork + ~2,970‚Äëline Metal renderer as an NSView subview under GPUI.
- **Reusable: renderer + full input stack ‚Äî *conditional on the Phase‚Äë0 seam passing*, not an unconditional ~100 %.** *If* first‚Äëresponder/IME/focus arbitration between GPUI's focus system and the embedded terminal routes correctly through a **live responder chain**, then the renderer (sub‚Äëline smooth GPU scroll, selection‚Äëacross‚Äëeviction, triple‚Äëbuffered pacing, dual atlases, kitty graphics/sixel) **and** the entire input stack baked into that NSView (kitty keyboard, legacy VT, `NSTextInputClient`/IME, VT mouse, copy/paste, drag‚Äëand‚Äëdrop) ride along with no rebuild. **That "if" is load‚Äëbearing.** The embedding spike used a Swift **stub** view and dispatched events by **invoking overridden selectors directly**, *not* through a live windowserver responder chain (`spike-reuse-swiftterm` "whatFailed"). IME marked‚Äëtext and kitty‚Äëkeyboard correctness depend on the genuine responder chain, so **the reuse percentage is unproven until Phase‚Äë0 drives real OS events through a live key window with the real SwiftTerm view.**
- **Interop cost (spike‚Äëmeasured):** the NSView host mechanics are a solved ~150‚Äëline problem. The real *ongoing* cost is (1) a hand‚Äëwritten `@_cdecl` C‚ÄëABI shim over SwiftTerm's Swift‚Äënative API (feed/resize/font/scroll/copy/colors), (2) reverse‚ÄëFFI for its delegate callbacks (title/bell/OSC/sizeChanged/clipboard), and (3) a permanent **cargo + swiftc** two‚Äëtoolchain build. The Swift *runtime* is OS‚Äëprovided and ABI‚Äëstable (`otool` confirmed `/usr/lib/swift/libswiftCore.dylib`), so there is **no runtime‚Äëshipping concern.**
- **The two unproven seams:** GPUI draws its *whole* content view itself, so (a) a foreign terminal NSView must sit at the right z‚Äëorder with GPUI rendering transparent over it, and (b) the responder chain/IME/focus must route correctly through a live key window ‚Äî *plus* the cross‚Äëwindow `setUseMetal` off‚Üíon rebind on tear‚Äëoff. **Path B ‚Äî NATIVE (fallback): `alacritty_terminal` + a GPUI‚Äënative `TerminalView`.** Zero reuse of the Metal renderer, but because **GPUI owns GPU glyph rasterization** you build a `TerminalView` on GPUI's text pipeline (as Zed/Paneflow do), *not* a Metal engine from scratch. **objc2‚Äëhybrid ‚Äî NATIVE‚ÄëCHROME fallback (¬ß4):** keep the SwiftTerm renderer, **no z‚Äëorder seam** (objc2 owns the view tree), but hand‚Äëwrite the chrome imperatively. The right fallback when Phase‚Äë0 fails *on the seam specifically* rather than on FPS.

### Why there's no fork ‚Äî the mechanisms (and the one boundary)

Every SwiftTerm fidelity feature maps onto GPUI's **fixed‚Äëbut‚Äërich** public primitive set ‚Äî quads, shadows, MSAA paths (box‚Äëdrawing/powerline), wavy underlines, **monochrome glyph sprites** (subpixel‚Äëpositioned, per‚Äëglyph transform), **polychrome sprites** (images/emoji, clipped, corner‚Äëradii, grayscale, opacity), and a **content‚Äëaddressed atlas** that takes arbitrary bytes ‚Äî all reachable through public `Window::paint_*` and the `canvas()`/custom‚Äë`Element` seam. **Zed's own production terminal is the existence proof**: it paints every cell through `text_system().shape_line(...).paint()` + `paint_quad(fill(...))` with **zero renderer fork**.

The **one** genuine fork boundary is a **bespoke Metal shader/pipeline** ‚Äî GPUI exposes no public hook to register one. That only flips the call to Path A *if* some specific SwiftTerm effect (e.g. a particular subpixel/gamma/blend AA technique) has **no** expression in the fixed primitive set. Because current GPUI already ships `SubpixelSprite` LCD‚ÄëAA, the fixed set is **very likely** sufficient ‚Äî but this is the lone unverified **pixel‚Äëcomparison** that a live AA/gamma side‚Äëby‚Äëside must close. *(‚öÝÔ∏è Audit: the `SubpixelSprite` support for this sentence is void ‚Äî dead code on macOS ‚Äî but main independently added default‚Äëon fg‚Äëluminance smoothing dilation to the mac rasterizer, so upstream does evolve this surface. The pixel comparison remains owed; see the scorecard.)*

### The non‚Äëfork costs Path B must still budget (orthogonal to the renderer)

These are real, but none touches GPUI's GPU text/atlas core:

### The single‚Äëdrawable option ‚Äî feasible only as an *additive* GPUI fork

Its defining premise ("single GPUI drawable + proven renderer + **no** fork") is **refuted** at the source level (closed renderer, no custom‚Äëshader hook, YUV‚Äëonly surface). **But** the strongest form ‚Äî an **additive sibling terminal pipeline** (a new `PrimitiveBatch` variant + `Scene` field + `draw_` method + `.metal` functions + a pipeline in `MetalRenderer::new`) that reuses GPUI's command buffer, drawable, and present clock and carries its **own** atlas ‚Äî does **not** touch GPUI's hardest subsystem (`text_system`/`open_type`/`metal_atlas`). So it is **materially cheaper than the feared "fork the GPU text core,"** and architecturally it would beat **both** A (one present clock ‚Äî no dual‚ÄëMetal co‚Äëpacing tax, no objc2 seam, no Swift FFI) and B (ports SwiftTerm's proven cell/atlas/sub‚Äëline‚Äëscroll technique ‚Äî no fidelity‚Äëregression risk). It does **not auto‚Äëdominate**, because it trades those for a **permanently‚Äëmaintained `gpui_macos` fork** ‚Äî squarely on the Claude‚Äëmaintainability hard gate. **Worth a cheap parallel spike** before final commitment; not a prerequisite for choosing B over A.

### Decision‚Äërule outcome and what remains

**Path B (provisional ‚Üí now strongly favored on source).** All six verified lanes converge that the ¬ß11 rule's load‚Äëbearing condition is **met**: SwiftTerm‚Äëclass fidelity is reachable on GPUI's **public** primitive/atlas/text API with **no** fork of `text_system`/`open_type`/`metal_atlas`, and Zed's own terminal proves the text path. So the call points to **Path B**, not Path A. It stays *provisional* only on the items source genuinely cannot settle ‚Äî all of which need a **built prototype + live‚ÄëGPU/display measurement** (this research pass was deliberately source‚Äëonly):

1. **Sustained burst FPS + present pacing** of sub‚Äëline scroll on a *built* single‚Äëstack GPUI‚Äënative terminal (vs SwiftTerm and vs Path A's Phase‚Äë0 p95~31 ms). *The clock/present question.*
2. **Keystroke‚Äëto‚Äëglyph latency** on the same prototype.
3. **AA/gamma/subpixel pixel comparison** vs SwiftTerm ‚Äî the **one** path that, if it needs a bespoke blend/shader, forces a fork and flips toward Path A.
4. **kitty/sixel terminal‚Äëgrid compositing** ‚Äî per‚Äëcell z‚Äëinterleave + clip + **atlas pressure** for large/**animated** images at refresh (the polychrome atlas was not built for video‚Äëin‚Äëterminal).
5. **Clean pty‚Äëstream DCS/APC tap** that splits sixel+kitty out without desyncing `vte`.
6. **kitty‚Äëkeyboard keyCode recovery** via objc2 side‚Äëchannel (avoid a GPUI patch); verify GPUI's keystroke normalization doesn't corrupt multi‚Äëlayout alternate‚Äëkey reporting.
7. **Selection‚Äëanchor survival across scrollback eviction** in the VT/`Selection` model.
8. **Single‚Äëdrawable additive‚Äëfork spike** ‚Äî build the minimal sibling pipeline, confirm one shared present clock with no added latency, and quantify fork merge‚Äëconflict cost against N months of Zed `main` history (the maintainability gate).

**Net:** the architecture decision is **de‚Äërisked from "feasibility unknown" to "favored, pending live measurement."** Items 1‚Äì4 are one focused build‚Äëand‚Äëmeasure session on the existing `spikes/phase0-poc/` harness; item 8 is the parallel "could‚Äëdominate‚Äëboth" probe.

### ‚úÖ LIVE MEASUREMENT ‚Äî the single‚Äëstack GPUI‚Äënative terminal locks 60 fps (2026‚Äë06‚Äë27)

**Built and measured remaining‚Äëitem #1 (the headline live unknown).** New runnable PoC `spikes/phase0-poc` binary **`gpui-term`** (`src/gpui_term.rs`, ~470 LOC): a **single‚Äëstack, GPUI‚Äënative terminal** ‚Äî an `alacritty_terminal` VT core rendered through GPUI's **public** paint API (`window.text_system().shape_line(...).paint()` for glyphs + `paint_quad` for cell backgrounds), inside GPUI's own **one** `CAMetalLayer`. **No SwiftTerm, no objc2 embed, no second Metal layer** ‚Äî literally the Path‚ÄëB architecture. It **reuses `harness.rs` verbatim** (same mach clock, FPS reducer, memory sampler, and synthetic Claude‚Äëstream workload as the Path‚ÄëA PoC), so the measurement *machinery* is identical to ¬ß10's. **‚öÝÔ∏è Audit:** the delivered workload was **not** identical ‚Äî the B side's per‚Äëframe byte budget is hard‚Äëcoded **2√ó higher** (`bytes_per_sec/60` vs `/120`; effective throughput 2.1‚Äì4√ó higher) into a ~2.2‚Äì2.5√ó‚Äëcell grid (120√ó40 full‚Äëre‚Äëshape vs A's damage‚Äëtracked 80‚Äì90√ó24), with no concurrent animated chrome, and the SwiftTerm side of ¬ß10 was release‚Äëbuilt while both Rust sides ran debug. The bias runs **conservative in B's favor** (heavier load, still locked 60), but magnitudes are not directly comparable. ### ‚úÖ AA/gamma comparison vs SwiftTerm ‚Äî the lone could‚Äëflip gate, resolved (2026‚Äë06‚Äë27)

**Method.** Rather than eyeball screenshots (confounded by font/size/scaling), this compares the **actual glyph‚Äëblend source on both sides**: GPUI's `monochrome_sprite_fragment` + sprite pipeline blend state (`gpui-0.2.2/src/platform/mac/shaders.metal:645‚Äë660`, `metal_renderer.rs:1244‚Äë1247`) vs SwiftTerm's text shaders + pipeline + rasterizer + atlas (`SwiftTerm/Sources/SwiftTerm/Apple/Metal/{Shaders.metal, MetalTerminalRenderer.swift, CoreTextGlyphRasterizer.swift, GlyphAtlas.swift}` + `TextCompositionCurve.swift`).

**Finding ‚Äî the fundamental text stack is identical; one optional curve is the only divergence.**
- **Rasterization:** both rasterize glyphs via **CoreText into an A8 grayscale atlas** (color/emoji into a separate BGRA atlas) ‚Äî the *same* model. (`CoreTextGlyphRasterizer.swift:35‚Äë63`, `GlyphAtlas.swift:20‚Äë41` ‚Üî GPUI's mono‚ÄëA8 + polychrome‚ÄëRGBA.)
- **Blend:** both use **premultiplied "over"** ‚Äî GPUI `color.a *= sample.a; return color` with `src=SourceAlpha/One, dst=OneMinusSourceAlpha`; SwiftTerm `src=.one (premultiplied), dst=.oneMinusSourceAlpha` (`MetalTerminalRenderer.swift:2828‚Äë2834`). **SwiftTerm uses NO LCD subpixel and NO dual‚Äësource blending** ‚Äî so GPUI's plain coverage path already matches it, and GPUI's **4√ó subpixel‚ÄëX position variants exceed** SwiftTerm's integer‚Äësnapped single‚Äëbitmap glyphs (`MetalTerminalRenderer.swift:15‚Äë19`).
- **The one bespoke element:** SwiftTerm's default‚Äëon‚ÄëmacOS grayscale shader `terminal_text_fragment_gray` (`Shaders.metal:100‚Äë113`) applies a **per‚Äëcell, luminance‚Äëaware gamma + contrast remap of coverage** ‚Äî `mix(coverage, pow(coverage, 1/1.7), mixFactor) * 1.30`, keyed on **both** fg luminance **and** per‚Äëcell bg luminance (Kitty's `text_composition_strategy 1.7 30` ‚âà Apple‚ÄëTerminal stem‚Äëdarkening). Because the same atlas texel is reused across cells with different fg/bg, it **cannot be baked into the atlas**, and GPUI exposes **no public hook** to inject `pow`/`mix`/contrast or per‚Äëcell luminance ‚Äî so GPUI's fixed primitive set **cannot reproduce it**.

**Verdict ‚Äî does NOT hard‚Äëflip to Path A; it isolates a bounded product decision.** Per the ¬ß11 decision rule (Path A only if matching fidelity forces a fork of GPUI's **text/atlas core**), this stays **Path B**: the core text stack matches with **zero** fork, and the lone gap is a single **optional, ~14‚Äëline fragment‚Äëshader tone curve** that (a) is **tunable to `.identity`**, where it reduces to plain coverage **byte‚Äëidentical to GPUI**, and (b) if wanted, is the **additive‚Äëshader‚Äëpatch class** (add one curve to `monochrome_sprite_fragment`), **not** a fork of `text_system`/`open_type`/`metal_atlas` ‚Äî exactly the cheap additive fork ¬ß12's single‚Äëdrawable analysis flagged, and current GPUI `main` already evolves this shader set (it added `SubpixelSprite`). # Addendum ‚Äî backfilled candidate (failed in workflow)

## Slint + alacritty_terminal (backfilled evaluation)

> Note: this candidate's deep-dive agent failed (its structured deep-dive exceeded the StructuredOutput schema-retry cap ‚Äî 5 schema-invalid attempts) in the main workflow, so Slint is absent from the workflow's head-to-head synthesis table. This section was researched out-of-band and is merged in here. Its conclusion (lower-middle of the Rust shortlist) means its absence did not change the top recommendation.

**Overall verdict.** A technically *native-efficient*, genuinely cross-platform Rust stack whose terminal half is excellent but whose chrome half is the wrong tool for *this* rewrite's primary driver. The core problem is that Slint's UI is written in a **bespoke `.slint` DSL** that is exactly the kind of niche, data-poor framework the rewrite is trying to escape SwiftUI for ‚Äî so the AI-assistability win over SwiftUI is marginal ‚Äî and its GPLv3/commercial licensing plus alacritty's "you rebuild the renderer" shape add real cost. It is a *viable* but **lower-middle** Rust option for Nice: weaker than GPUI or a winit/objc2 DIY for this specific app.

### Scored dimensions

| # | Dimension | Score | Conf. | One-line rationale |
|---|-----------|:---:|:---:|---|
| D1 | AI-language (Rust) | **4/5** | high | Rust is a top-tier, very-well-represented language in Claude's corpus; borrow-checker/lifetime friction is the only drag, not knowledge. |
| D2 | AI-UI-framework (Slint + `.slint` DSL) | **2/5** | medium | ~23k-star lib with decent first-party docs, but the bespoke `.slint` markup DSL is niche with thin Stack Overflow / real-repo presence ‚Äî the same "data-poor framework" trap as SwiftUI. |
| D3 | Efficiency (mem/CPU) | **4/5** | medium | Compiled native toolkit, Skia/Metal GPU renderer, tiny runtime core; firmly native-class, nowhere near Electron. No public Slint-vs-SwiftUI RSS benchmark exists ‚Äî parity is inferred from architecture, not measured. |
| D4 | Styling (vibrancy/blur/chrome) | **2/5** | med-high | Slint draws its *own* widgets; no native NSVisualEffectView/vibrancy exposure ‚Äî true liquid-glass needs a hand-written unsafe AppKit bridge, and you fight Slint's window-property overriding. |
| D5 | Terminal (alacritty_terminal) | **3/5** | medium | `alacritty_terminal` is a proven, excellent state engine (Zed/ZTerm/Termy), but ships no renderer; high-perf path is a from-scratch wgpu renderer imported via Slint's experimental `Image::try_from<wgpu::Texture>`. |
| D6 | Windowing (multi-window + tear-off) | **3/5** | med-low | Real multi-window since 1.7 (winit-backed); pane tear-off is feasible because the terminal is *data* (move the `Term` to a new window, no NSView to reparent), but Slint aggressively overrides window props (no first-class custom window mgmt). |
| D7 | Maturity + licensing | **3/5** | high | Stable 1.x, frequent releases (1.17, Jun 2026), commercial backing, production use ‚Äî but GPLv3 / royalty-free-with-strings / paid-commercial, a real adoption cost vs MIT/Apache peers; desktop window APIs still unstable. |
| D8 | Linux (bonus) | **5/5** | high | First-class X11/Wayland via winit; `alacritty_terminal` is fully cross-platform. Strong free win. |

### Styling verdict
Slint renders every widget itself through Skia/FemtoVG into a single winit-owned root `NSView`; it has no native-material element and no public API for NSVisualEffectView vibrancy, rounded native chrome, or custom traffic-lights. What *is* supported: transparent/non-opaque window backgrounds (PR #2649) and an optional native macOS menu bar. Everything beyond that ‚Äî blur/vibrancy, hiding/replacing the titlebar, traffic-light placement ‚Äî is an unsupported manual bridge: reach the raw window handle through the private `i-slint-backend-winit` crate, cast to `NSWindow`/`NSView`, and drive AppKit with `objc2`/`cocoa` yourself (community recipe in discussion #5710; custom-titlebar is still only proposal #2521; hide-titlebar is a private-API workaround in #4284). Worse, Slint is documented to override window properties you set (issue #11001, "impossible to get real full screen on macOS"), so you partly fight the framework to get bespoke chrome. Net: Nice's liquid-glass look is *achievable* only by writing roughly the same AppKit glue a DIY objc2 approach needs ‚Äî so Slint buys you little here while still constraining you.

Sources: discussion #5710 ¬∑ PR #2649 ¬∑ issue #2521 ¬∑ discussion #4284 ¬∑ issue #11001 ¬∑ Slint MenuBar docs

### Biggest risks
1. **Marginal AI win over SwiftUI (kills the primary driver).** The `.slint` DSL is as niche/data-poor as SwiftUI; Claude leans on first-party docs, not a deep public corpus. D1 (Rust) is strong, but the rewrite's value lives in D2, and D2 is weak.
2. **Styling is DIY anyway.** You write nearly the same unsafe AppKit/objc2 bridge for vibrancy/chrome as a from-scratch winit approach ‚Äî without the full control that approach gives.
3. **High-perf terminal rendering rides an experimental surface.** wgpu-texture import (`Image::try_from<wgpu::Texture>`) landed in 1.12 behind an `unstable-wgpu-NN` feature, version-pinned to specific wgpu releases ‚Äî a moving target for a long-lived app.
4. **Throws away the proven Metal work.** alacritty path = rebuild smooth-scroll, selection-across-stream, and resize-coalescing on a new renderer; the SwiftTerm investment is sunk.
5. **Licensing.** GPLv3 forces open-sourcing a proprietary app; the royalty-free desktop license carries Slint's own terms; commercial is paid ‚Äî a standing constraint vs permissive peers.

### Adversarial self-check (load-bearing claims vs primary sources)
- "Slint has no native vibrancy; you must bridge AppKit yourself." ‚Äî **Supported.** Only community raw-handle/`cocoa` recipes exist; PR #2649 adds background transparency only, no NSVisualEffectView.
- "`.slint` DSL is a niche, data-poor corpus." ‚Äî **Supported (with nuance).** Repo healthy at ~23k stars and Rust well-covered, but the *DSL specifically* has little third-party corpus; "data-poor" is a reasoned inference, not a measured count (confidence medium).
- "alacritty_terminal provides no renderer; you rebuild it." ‚Äî **Supported.** Docs scope it to grid/term/selection/pty/parser; Zed/ZTerm/Termy each pair it with their own GPU renderer.
- "Native efficiency, far from Electron, ~parity with SwiftUI." ‚Äî **Partly uncertain.** Native-class/non-Electron well-supported; the "<300 KiB runtime" figure is the embedded *core*, not a desktop app's RSS, and no Slint-vs-SwiftUI benchmark was found (D3 confidence capped at medium).
- "wgpu texture import is experimental." ‚Äî **Supported.** Exposed via `unstable-wgpu-NN` feature gates, wgpu-version-pinned.
- "Real multi-window exists but custom window mgmt is constrained." ‚Äî **Supported.** Multi-window since 1.7; window-property-override limitation documented in #11001.
- "Embedding the existing SwiftTerm NSView under Slint isn't viable." ‚Äî **Uncertain/likely-true.** Slint owns a single winit root NSView and exposes no AppKit-view-host element; foreign-NSView subview insertion is undocumented and unsupported.

### Ranking note
Likely **below GPUI and below a winit/objc2 DIY** for Nice, and roughly **on par with or just under Iced**: GPUI is the proven Rust-terminal path (Zed/ZTerm/Termy = GPUI + `alacritty_terminal`) with permissive Apache licensing; a winit/objc2 DIY gives true vibrancy and could even re-host the existing SwiftTerm Metal NSView (preserving the Metal work); Iced avoids a bespoke DSL (pure-Rust UI = better Claude leverage on the framework) at some cost to native styling. Slint's bespoke `.slint` DSL (weak D2) + GPL/commercial licensing + sunk Metal work are what push it down the Rust shortlist.
